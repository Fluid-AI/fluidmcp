React Architecture

9 pages, 11 custom hooks, one API singleton. Here is how it all fits together.

🗺
9 Pages
Each page is a route component in src/pages/. Pages are thin — they own layout and local UI state but delegate all data to hooks.
🪝
11 Custom Hooks
All API calls and business logic live in src/hooks/. No Redux or Zustand — each feature owns its state in a dedicated hook.
🔗
One API Client
services/api.ts is the only file that calls fetch(). Hooks import this singleton and call its typed methods.
📌
Reading this tutorial: Lesson 3 covers all 9 pages briefly. Lessons 4–6 go deep on the three most important ones. Lesson 7 covers the hook patterns shared across the whole codebase.

│ ├── ui/ # shadcn/ui primitives (Button, Dialog…)
│ ├── form/ # Dynamic JSON Schema form renderers
│ └── result/ # Tool result viewers (JSON/Table/Text/MCP)
│
├── services/ # Non-React utilities
│ ├── api.ts # Singleton ApiClient — only fetch() caller
│ └── toolHistory.ts # localStorage tool execution history

└── types/ # TypeScript interfaces
├── server.ts # Server, Tool, EnvVar
└── llm.ts # LLMModel, ReplicateModel, ChatMessage
🧠
Rule of thumb: pages = what the user sees. hooks = the logic and data behind it. api.ts = the only wire to the backend. If you are debugging a data issue, start in the hook. If you are debugging a UI issue, start in the page.

All Pages at a Glance

9 routes, defined in App.tsx. Unknown paths fall back to /.

📍
Always enter at /ui. The backend mounts the frontend at /ui — going to /servers or /llm/models directly in the browser will not work. Start at /ui and React Router handles all navigation from there.

Codespace: https://<name>-8099.app.github.dev/ui
Local: http://localhost:8099/ui
Railway: https://<your-app>.railway.app/ui
PageRouteHook(s)Purpose
Dashboard/ or /serversuseServersServer grid with search, filter, sort, paginate. Quick start action.
Status/statususeServers, useActiveServerFilteringRunning/starting servers only. Quick stop/restart.
ServerDetails/servers/:iduseServerDetails, useServerPollingThree tabs: Tools list, live Logs, Environment variables.

A note on Documentation and LLMPlayground

Documentation has no API calls — all content is hardcoded. It uses useLocation() only to read the URL hash and scroll to the right section on load.

LLMPlayground calls apiClient.chatCompletion() directly from the component (no custom hook) because it is essentially a one-shot stateful form — no list to manage, no polling needed.

Dashboard

src/pages/Dashboard.tsx — the landing page. Shows all MCP servers as cards.

What it does
  • Fetches all servers on mount via useServers
  • Client-side filter (all / running / stopped / error), search by name, sort (name A–Z, name Z–A, status, recent)
  • Pagination — 9 cards per page; scrolls back to list top on page change
  • "Start" triggers startServer(id) with toast feedback; actionState prevents double-clicks
  • "View Details" navigates to /servers/:id
  • Skeleton loading state while data loads; ErrorMessage + retry on error
  • AOS (Animate On Scroll) on each card for entrance animations

Hook wiring

Dashboard
└─ useServers()
├ servers[] → filtered → sorted → paginated → ServerCard[]
├ activeServers[] → subtitle "N running"
├ loading → skeleton grid
├ error → ErrorMessage + retry
├ startServer(id) → POST /api/servers/:id/start → refetch
└ refetch() → "Refresh" button

Key local state

typescript
const [searchQuery, setSearchQuery] = useState("");
const [sortBy, setSortBy]    = useState<'name-asc'|'name-desc'|'status'|'recent'>('name-asc');
const [filterBy, setFilterBy] = useState<'all'|'running'|'stopped'|'error'>('all');
const [currentPage, setCurrentPage] = useState(1);

// Prevents double-click start while request is in flight
const [actionState, setActionState] = useState<{
  serverId: string | null;
  type: 'starting' | 'stopping' | null;
}>({ serverId: null, type: null });
💡
Filtering is client-side. The hook fetches all servers once; Dashboard filters them in memory. This is fine for typical server counts (<100). No server-side filtering on Dashboard.

Manage Servers

src/pages/ManageServers.tsx — the CRUD control panel. Intentionally separate from Dashboard.

What it does
  • Table layout (not cards) — columns: name/ID, command, status, Edit/Delete
  • Fetches all servers including disabled ones (enabled_only: false)
  • "Add Server" opens a modal with two tabs: Manual Config and Clone from GitHub
  • Editing a running server is blocked — must stop it first
  • Delete opens DeleteConfirmationModal with two choices: soft-delete (disable) or hard delete
  • "Deleted" toggle reveals soft-deleted servers for admin recovery
  • Search by name or ID, filter by status, paginated at 10 per page

Hook wiring

ManageServers
└─ useServerManagement(showDeleted)
├ servers[] → filtered → paginated → table rows
├ createServer(config) → POST /api/servers → refetch
├ updateServer(id, config) → PUT /api/servers/:id → refetch
├ deleteServer(id) → DELETE /api/servers/:id → refetch
└ refetch(showDeleted) → called when toggle changes

The edit guard

typescript
const handleEdit = (server: Server) => {
  if (server.status?.state === 'running') {
    alert('Cannot edit running server. Stop it first.');
    return;
  }
  setEditingServer(server);
  setShowForm(true);
};

Two ways to add a server

⚙️
Manual Config
Fill in name, command, args, env vars. Uses ServerForm component. Calls POST /api/servers.
📥
Clone from GitHub
Provide a GitHub repo URL + token. Uses CloneFromGithubForm. Calls POST /api/servers/from-github which clones and extracts metadata.

LLM Models

src/pages/LLMModels.tsx — the Dashboard equivalent for AI models.

What it does
  • Grid of LLM model cards — same layout pattern as Dashboard
  • Supports three model types: process (vLLM/Ollama local), replicate (cloud API)
  • Debounced search via useDebounce(searchQuery, 300) — filters on model ID
  • Filter by: all / running / stopped / healthy / unhealthy / process / replicate
  • Sort by: name, health score, uptime
  • Filter logic is useMemo-memoized (Dashboard does it inline)
  • "Add Model" opens LLMModelForm modal; "Try LLMs" navigates to playground
  • Shows summary stats: total, running, healthy counts

Hook wiring

LLMModels
├─ useLLMModels()
├ models[] → filtered (useMemo) → sorted → paginated → LLMModelCard[]
├ runningModels[], healthyModels[] → subtitle stats
├ loading, error → skeleton / error state
└ refetch() → called after create/update
└─ useDebounce(searchQuery, 300)
└ debouncedSearchQuery → passed into useMemo filter

Why useDebounce here but not on Dashboard?

LLMModels filters by model ID which may have many similar-looking strings — debouncing prevents re-filtering on every keystroke. Dashboard filters by server name which is short and the list is typically smaller, so inline filtering is fine there.

Creating a model (direct apiClient call)

typescript
const handleCreateModel = async (config: ReplicateModelConfig) => {
  const toastId = showLoading('Creating model...');
  try {
    await apiClient.createLLMModel(config);   // POST /api/models/register
    showSuccess('Model created successfully', toastId);
    refetch();
  } catch (err) {
    showError(`Failed: ${err.message}`, toastId);
    throw err;
  }
};

Hooks — How It All Wires Up

Three patterns repeat across every hook in the codebase. Learn these and you will understand them all.

Pattern 1: AbortController + isMountedRef

Prevents state updates after a component unmounts (avoids React warnings and memory leaks):

typescript — useServers.ts (representative example)
const isMountedRef = useRef(true);
const abortControllerRef = useRef<AbortController | null>(null);

const fetchServers = useCallback(async () => {
  abortControllerRef.current?.abort();           // cancel previous in-flight request
  const controller = new AbortController();
  abortControllerRef.current = controller;

  setLoading(true);
  try {
    const response = await apiClient.listServers({ signal: controller.signal });
    if (isMountedRef.current) setServers(response.servers);
  } catch (err) {
    if (!controller.signal.aborted && isMountedRef.current)
      setError(err.message);
  } finally {
    if (isMountedRef.current) setLoading(false);
  }
}, []);

useEffect(() => {
  isMountedRef.current = true;
  fetchServers();
  return () => {
    isMountedRef.current = false;
    abortControllerRef.current?.abort();
  };
}, [fetchServers]);

Pattern 2: Refetch after every mutation

Every write (start/stop/create/update/delete) calls the fetch function at the end to keep UI in sync:

typescript — useServerManagement.ts
const createServer = useCallback(async (config: any) => {
  try {
    await apiClient.addServer(config);
    showSuccess(`Server created`);
    await fetchServers();   // always re-sync after write
    return true;
  } catch (err) {
    showError(err.message);
    return false;
  }
}, [fetchServers]);

Pattern 3: useCallback on fetch to avoid infinite loops

The fetch function is wrapped in useCallback so its reference is stable. This prevents the useEffect([fetchData]) from re-running on every render:

typescript
// fetchServers is stable — only created once (no deps)
const fetchServers = useCallback(async () => { ... }, []);

// So this effect runs only once on mount
useEffect(() => {
  fetchServers();
}, [fetchServers]);
✅
Summary: Every hook follows the same skeleton: useCallback on fetch → useEffect calls it on mount / cleans up on unmount → mutations call fetch at the end. Once you see it once you will recognise it everywhere.

Adding a new page — the pattern to follow

  1. Create src/pages/MyPage.tsx — thin component, local UI state only
  2. Create src/hooks/useMyFeature.ts — fetch + mutations using the three patterns above
  3. Add typed methods to src/services/api.ts if you need new endpoints
  4. Add <Route path="/my-page" element={<MyPage />} /> in App.tsx
  5. Add nav link in Navbar.tsx, run npm run build, restart backend
🎉
You are oriented. You know every page, you can read any hook, and you know where to add new features. The architecture is consistent — once you understand one page+hook pair, you understand them all.
