Welcome to the FluidMCP Frontend

A React SPA served by the FastAPI backend — your UI for managing MCP servers, running tools, and chatting with LLM models.

⚡
React + TypeScript + Vite
Modern SPA with type safety and fast builds. No separate frontend server in production — Python serves the built dist/ folder.
🏗
9 Pages, 11 Custom Hooks
Each page is a route. All data fetching and state lives in custom hooks — no Redux or Zustand.
🔗
Single API Client
services/api.ts is the only file that calls fetch(). All 30+ typed methods live there as a singleton.
📌
Where it lives: fluidmcp/frontend/ inside the monorepo. Built output goes to fluidmcp/frontend/dist/ which the Python backend serves at /ui. Note: /docs is the Swagger API docs (FastAPI built-in), not the frontend.

What you can do in the UI

  • Browse all MCP servers and their run status (Dashboard)
  • Add, edit, delete and start/stop servers (Manage Servers)
  • Execute MCP tools with auto-generated forms (Tool Runner)
  • Manage and test LLM models with a chat interface (LLM Models + Playground)
  • View server logs and environment variables (Server Details)

How they connect

index.html → Vite entry point, loads main.tsx
main.tsx → ReactDOM.createRoot, mounts <App />
App.tsx → React Router routes, Navbar + Footer layout
pages/*.tsx → Route components, use custom hooks for data
hooks/*.ts → useState + useEffect + apiClient calls
services/api.ts → fetch() to FastAPI backend at port 8099
✅
No global state library. State is co-located with the feature. If two pages need the same data they each call their own hook instance — or the hook could be lifted into a context (but that has not been needed yet).

Local Development Setup

Get the frontend running on your machine in four commands.

📋
Prerequisites: Node.js 18+ and npm installed. Python backend set up (pip install -e . from the repo root).

Step-by-step

  1. 1
    Install npm dependencies Run this once (or after pulling dependency changes).

  1. 2

  1. 3
    Start the backend (serves the frontend too) The Python backend mounts dist/ and serves it at the root URL.

Environment variable

One optional Vite env var controls where the frontend points for API calls. You usually only need this when running the Vite dev server (npm run dev) separately from the backend:

env (fluidmcp/frontend/.env.local)
VITE_API_BASE_URL=https://<codespace-name>-8099.app.github.dev

If not set, the frontend falls back to window.location.origin — correct when frontend and backend share the same origin (the normal production setup).

⚠️
No hot-reload when using the built dist. After changing frontend code run npm run build and refresh. For active development use npm run dev (Vite on port 5173) which hot-reloads instantly and proxies API calls to the backend via vite.config.ts.

Frontend ↔ Backend Communication

All HTTP calls go through one singleton: services/api.ts. Here is how it works.

The ApiClient singleton

typescript — services/api.ts (simplified)
class ApiClient {
  private baseUrl: string;

  constructor() {
    // Use VITE_API_BASE_URL env var, or fall back to same origin
    this.baseUrl = import.meta.env.VITE_API_BASE_URL || window.location.origin;
  }

  private async request<T>(endpoint: string, options?: RequestInit): Promise<T> {
    const response = await fetch(`${this.baseUrl}${endpoint}`, {
      ...options,
      headers: { 'Content-Type': 'application/json', ...options?.headers }
    });
    if (!response.ok) throw new ApiHttpError(await response.text(), response.status);
    return response.json();
  }

  // Typed public methods (30+ total)
  listServers()                          { return this.request('/api/servers'); }
  startServer(id: string)               { return this.request(`/api/servers/${id}/start`, { method: 'POST' }); }
  runTool(serverId, toolName, params)    { return this.request(`/${serverId}/mcp/tools/call`, { method: 'POST', body: JSON.stringify(params) }); }
  chatCompletion(payload)                { return this.request('/api/llm/v1/chat/completions', { method: 'POST', body: JSON.stringify(payload) }); }
}

// Exported as a singleton — never instantiate again
export default new ApiClient();

The call chain: page → hook → api

Dashboard.tsx → const { servers, startServer } = useServers()
useServers.ts → apiClient.listServers() / apiClient.startServer(id)
api.ts → fetch('/api/servers') → FastAPI backend
← JSON response → typed result → useState → re-render

Error handling pattern

typescript
// ApiHttpError carries the HTTP status code
export class ApiHttpError extends Error {
  constructor(public message: string, public status: number) {
    super(message);
  }
}

// In a hook — catch and surface to UI state
try {
  const data = await apiClient.listServers();
  setServers(data.servers);
} catch (err) {
  if (err instanceof ApiHttpError) {
    setError(`HTTP ${err.status}: ${err.message}`);
  }
}

Request cancellation

Hooks use AbortController to cancel in-flight requests when a component unmounts or a new request supersedes the old one:

typescript
const abortRef = useRef<AbortController | null>(null);

const fetchData = async () => {
  abortRef.current?.abort();              // cancel previous
  abortRef.current = new AbortController();
  await apiClient.someMethod({ signal: abortRef.current.signal });
};

useEffect(() => {
  fetchData();
  return () => abortRef.current?.abort(); // cleanup on unmount
}, []);

Build & Deploy

FluidMCP is deployed on Railway. The frontend ships inside the same container as the backend — no separate frontend deployment.

Railway (primary deployment)

Railway auto-detects the Dockerfile in the repo root, builds the full app, and deploys it. You never run npm run build manually for Railway — the Dockerfile handles it.

Environment variableHow to set it
MONGODB_URIAuto-provided by the Railway MongoDB service
FMCP_BEARER_TOKENMust set manually. Railway containers are ephemeral — without this a new token is generated on every restart, breaking auth. Generate: openssl rand -hex 32
PORTAuto-assigned by Railway
📖
Go deeper on deployment: see docs/RAILWAY_DEPLOYMENT.md for the full step-by-step setup, API management after deploy, MongoDB connection behaviour, and troubleshooting. The backend docs also cover health checks and persistence modes.

Docker (alternative / self-hosted)

The same Dockerfile can be used outside Railway for self-hosted deployments:

bash
docker build -t fluidmcp:latest .
docker run -d -p 8099:8099 \
  -e MONGODB_URI="mongodb://host.docker.internal:27017" \
  -e FMCP_BEARER_TOKEN="your-token-here" \
  fluidmcp:latest
# UI at http://localhost:8099/ui

How the backend serves the frontend

fluidmcp/cli/api/frontend.py mounts the dist/ folder as a Starlette StaticFiles mount under /ui and adds a catch-all for /ui/* that returns index.html — enabling React Router’s client-side routing to work correctly.

Request pathServed by
/ui and /ui/*React SPA (dist/index.html + Vite assets)
/api/*FastAPI route handlers
/docsSwagger UI (FastAPI built-in)
/healthFastAPI health endpoint
