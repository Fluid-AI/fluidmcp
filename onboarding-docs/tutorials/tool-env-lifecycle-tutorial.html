Architecture Overview

Lesson 1 of 6 ~8 min read

FluidMCP's UI and backend form a five-layer stack. Every user action — adding a server, running a tool, editing an env var — flows through all five layers. Understanding the stack makes the individual flows in later lessons easy to follow.

The Five Layers

React SPA (Browser) src/pages/ · src/hooks/ · src/services/api.ts
▼
FastAPI Gateway fluidmcp/cli/api/management.py
▼
ServerManager fluidmcp/cli/services/server_manager.py
▼
Persistence (MongoDB / In-Memory) fluidmcp/cli/repositories/database.py · memory.py
▼
MCP Subprocess npx / python / uvx — child process managed by ServerManager

Key Concepts

Flat config format

The REST API and ServerManager always work with a flat config dict: id, name, command, args, env at the root level. When saving to MongoDB, DatabaseManager nests these under mcp_config internally. This nesting never appears in API requests or responses.

Two storage layers for env vars

Every server has two separate env namespaces in MongoDB:

LayerCollectionWhen writtenSurvives restart?
Server configfluidmcp_serversPOST /api/servers (add time)Yes — this is the template
Instance envfluidmcp_server_instancesPUT /api/servers/:id/instance/envYes — persisted, triggers auto-restart

Transport types

MCP servers communicate with FluidMCP over one of two transports. The transport is set in metadata.json (or defaults to stdio):

TransportHow it worksInit check
stdio (default)JSON-RPC over stdin/stdout pipesJSON-RPC initialize handshake (30 s timeout)
sse / httpHTTP server (uvicorn/starlette)TCP port poll via asyncio.open_connection()

Server Lifecycle

Lesson 2 of 6 ~12 min read

This lesson traces how an MCP server moves from first registration through Start → Running → Stop, and what happens in each layer along the way.

Adding a Server

The user fills the "Add Server" form in ManageServers.tsx and clicks Save. The hook calls apiClient.addServer() which fires: 

# POST /api/servers
# Body is FLAT — command/args/env at the root, not nested
curl -X POST http://localhost:8099/api/servers \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "filesystem",
    "name": "Filesystem Server",
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
    "env": {},
    "restart_policy": "on-failure",
    "max_restarts": 3
  }'
⚠️
Flat format only The field is "id", not "server_id". Do not nest config under mcp_config — that is an internal MongoDB storage detail only. Sending nested format returns a 422 or silently drops the command.

What happens in the backend (management.py → add_server()):

  1. Sanitises input to prevent MongoDB injection
  2. Validates id and name are present
  3. Calls validate_server_config() — checks command against the allowlist (npx, node, python, python3, uvx, docker, deno, bun)
  4. Checks for duplicate id — raises HTTP 409 if already in DB or in-memory registry
  5. Calls DatabaseManager.save_server_config(config) — persists to MongoDB; falls back to in-memory on DB failure
  6. Registers config in ServerManager.configs[id] for immediate in-memory access
  7. Returns { "id": "...", "name": "...", "message": "..." } — status is implicitly stopped

Starting a Server

# POST /api/servers/:id/start
curl -X POST http://localhost:8099/api/servers/filesystem/start \
  -H "Authorization: Bearer $TOKEN"

What ServerManager.start_server() does:

  1. Loads flat config from manager.configs[id] (originally loaded from MongoDB)
  2. Merges any persisted instance env from fluidmcp_server_instances
  3. Spawns: subprocess.Popen([command] + args, stdin=PIPE, stdout=PIPE, env=merged_env)
  4. Initialization handshake — depends on transport:
    • stdio: writes JSON-RPC initialize to stdin → reads response (30 s timeout) → sends notifications/initialized
    • sse/http: polls TCP port via asyncio.open_connection() until uvicorn accepts connections
  5. Calls tools/list → caches tool schemas in fluidmcp_server_instances
  6. Records PID in manager.processes[id] and updates status to running
ℹ️
30-second init timeout If the MCP subprocess doesn't respond to the initialize handshake within 30 seconds, the start call returns HTTP 500. This is the most common cause of start failures for new or misconfigured servers.

Stopping a Server

# POST /api/servers/:id/stop
# Optional: ?force=true to send SIGKILL immediately (skip SIGTERM)
curl -X POST http://localhost:8099/api/servers/filesystem/stop \
  -H "Authorization: Bearer $TOKEN"

# Force kill (SIGKILL) — use when SIGTERM doesn't work
curl -X POST "http://localhost:8099/api/servers/filesystem/stop?force=true" \
  -H "Authorization: Bearer $TOKEN"

The shutdown sequence in ServerManager._stop_server_unlocked():

  1. Sends SIGTERM to the process (or SIGKILL if force=true)
  2. Waits up to 10 seconds for the process to exit
  3. If still running after 10 s, sends SIGKILL (force kill)
  4. Removes PID from manager.processes and updates status to stopped
⚠️
Authorization check on stop The stop endpoint checks that the requesting user is the same user who started the server (matched by token hash). A different user gets HTTP 403 unless no owner was recorded (backward-compatibility) or the server runs in anonymous mode.

Restarting a Server

# POST /api/servers/:id/restart
curl -X POST http://localhost:8099/api/servers/filesystem/restart \
  -H "Authorization: Bearer $TOKEN"

Restart is a sequential Stop → Start under the same operation lock. The config is re-read from manager.configs[id] (originally from MongoDB), so the server always restarts with the canonical stored config.

ℹ️
Instance env persists across restarts Env overrides written via PUT /api/servers/:id/instance/env are stored in fluidmcp_server_instances and are merged back in on every start. See Lesson 4 for details.

Tool Execution

Lesson 3 of 6 ~10 min read

This lesson traces what happens from the moment a user clicks "Run Tool" in the ToolRunner page all the way down to the MCP subprocess and back.

Navigation to ToolRunner

/ui
  → Dashboard
  → click "View Details" on a server card
  → ServerDetails (/servers/:id)
  → Tools tab
  → click "Run" next to a tool name
  → ToolRunner (/servers/:id/tools/:toolName)

On Mount

When ToolRunner.tsx mounts, useEffect fires two parallel fetches: 

# 1. Load server details (name, config)
GET /api/servers/:id

# 2. Load tool schemas for this server
GET /api/servers/:id/tools

The tool matching toolName from the URL is found in the returned array. Its inputSchema (JSON Schema) is passed to <JsonSchemaForm> which auto-generates form fields — one input per property, with type-appropriate widgets (text, number, checkbox, etc.).

Executing a Tool

When the user fills the form and clicks "Run Tool":

  1. handleSubmit(values) in ToolRunner.tsx calls useToolRunner.execute(values)
  2. Hook sets loading=true, records startTime = performance.now()
  3. Calls apiClient.runTool(serverId, toolName, args)
# POST /api/servers/:id/tools/:toolName/run
curl -X POST http://localhost:8099/api/servers/filesystem/tools/read_file/run \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "arguments": {
      "path": "/tmp/hello.txt"
    }
  }'

FastAPI → ServerManager → MCP Subprocess

Inside management.py:

  1. Validates bearer token (if secure mode)
  2. Looks up the server in ServerManager
  3. Calls server_manager.run_tool(server_id, tool_name, args)

Inside server_manager.run_tool():

  1. Acquires per-server threading.Lock to serialise concurrent tool calls
  2. Builds the JSON-RPC request: 
    {
  "jsonrpc": "2.0",
  "id": N,
  "method": "tools/call",
  "params": { "name": toolName, "arguments": args }
}
  3. Writes to process.stdin and flushes
  4. Reads one line from process.stdout (the JSON-RPC response)
  5. Releases the lock and returns the parsed response

The MCP subprocess response format:

{
  "jsonrpc": "2.0",
  "id": N,
  "result": {
    "content": [
      { "type": "text", "text": "file contents here" }
    ]
  }
}

Back in the Browser

  1. Hook calculates executionTime = (performance.now() - startTime) / 1000 in seconds
  2. Calls setResult(response), setLoading(false)
  3. Saves execution to localStorage via toolHistoryService.saveExecution()
  4. ToolRunner re-renders: layout switches to a 2-column grid — form on the left, result on the right

Result Format Detection (ToolResult.tsx)

ToolResult receives the raw HTTP response and picks a renderer automatically:

ConditionRenderer
Response has MCP content[] arrayMcpContentView — handles text / image / resource blocks
Response is an array of objectsTableResultView — sortable table
Response is a plain object or complex JSONJsonResultView — syntax-highlighted JSON
Response is a string or numberTextResultView — monospace text

Tool Execution History

toolHistoryService (src/services/toolHistory.ts) persists executions to localStorage under a key of serverId + toolName:

  • Max 50 entries per tool (LRU eviction)
  • Both successful and failed executions are stored
  • "Load Parameters" in the history drawer calls loadFromHistory(id) and re-populates the form
  • History survives browser sessions until localStorage is cleared

Error Path

If the tool call fails (network error, MCP error, timeout):

// useToolRunner catch block
setError(errorMessage)
setExecutionTime(duration)    // still recorded
setLoading(false)
toolHistoryService.saveExecution({ ..., success: false, error: errorMessage })

The right column renders <ToolResult error={executionError} /> and the history entry is marked with a red "✗ Failed" badge.

Environment Variables

Lesson 4 of 6 ~10 min read

The Environment tab in ServerDetails lets users view and update env vars for a running server. Understanding how env changes persist — and what happens when you save them — is critical for avoiding surprises.

Navigation to the Environment Tab

/ui
  → Dashboard
  → click "View Details" on a server card
  → ServerDetails (/servers/:id)
  → "Environment" tab

The Data Model

The backend never sends actual secret values to the frontend. Each env var is returned as metadata only: 

// src/types/server.ts
interface EnvMetadata {
  present:     boolean;        // Is a value set on the running instance?
  required:    boolean;        // Is this var required for the server to work?
  masked:      string | null;  // "****" if present, null if not set
  description: string;         // Help text shown in the UI
}
🔒
Values never leave the server The backend only ever returns "****" for set values. The actual secret is never transmitted to the browser — it cannot appear in DevTools, logs, or network traces.

Read Flow

When the Environment tab activates, useServerEnv(serverId) fires:

# GET /api/servers/:id/instance/env
curl http://localhost:8099/api/servers/filesystem/instance/env \
  -H "Authorization: Bearer $TOKEN"

The backend response:

{
  "API_KEY": { "present": true,  "required": true,  "masked": "****", "description": "Your API key" },
  "DEBUG":   { "present": false, "required": false, "masked": null,   "description": "Enable debug logging" }
}

The UI renders each key as a row with a [required] badge where applicable. The masked value **** is shown where a value is present.

Write Flow

When the user edits a value and clicks Save:

# PUT /api/servers/:id/instance/env
# Body: flat dict of key → new-value pairs
curl -X PUT http://localhost:8099/api/servers/filesystem/instance/env \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "API_KEY": "sk-new-value"
  }'

What the backend does in management.py → update_server_instance_env():

  1. Validates the env dict (security + DoS checks via validate_env_variables())
  2. Filters out placeholder values like "YOUR_API_KEY_HERE" or empty strings
  3. Calls manager.db.update_instance_env(id, filtered_env) — persists to MongoDB in fluidmcp_server_instances
  4. Checks if the server is currently running
  5. If running: automatically restarts the server to apply the new values
  6. If restart fails: rolls back the env change in MongoDB and returns HTTP 500
🚨
Saving env vars restarts a running server Unlike what the flow diagram comment suggests, saving env vars via PUT /instance/env does restart the server if it is currently running. This is intentional — env vars must be injected at process spawn time and can't be hot-patched into a running process. Plan accordingly in production environments.

Env Persistence Model

ActionWhere savedSurvives restart?
Env declared in POST /api/servers body fluidmcp_servers (config template) Yes — always used as the base
Env set via PUT /instance/env fluidmcp_server_instances (instance overrides) Yes — merged on every start, overrides template values

On every start_server(), both layers are merged: instance env takes precedence over the config template.

Required Fields

Fields marked required: true in the metadata show a badge in the UI. If a required var is not present, the UI surfaces it visually so the user knows the server may not function correctly — but the backend does not block start.

State & Error Patterns

Lesson 5 of 6 ~8 min read

This lesson covers how server status is tracked, common error scenarios and their HTTP codes, and how the UI stays current without requiring manual refreshes.

Server States

StateUI badgeWhat it means
stoppedStoppedNot running; Start button shown
startingStartingProcess spawned, init handshake in progress
runningRunningHandshake complete; tools available; Stop + Restart shown
errorErrorSpawn failed or init timed out; last_error is set

Status Polling

Dashboard cards call GET /api/servers on a short polling interval so status badges stay current. There is no WebSocket push — polling is intentional to keep the connection model simple. 

# Check a single server's status
GET /api/servers/:id/status

# List all servers with their current state
GET /api/servers

Error Cases

ErrorHTTP codeCause
Duplicate id on Add409Server with that ID already in DB
Missing command field400Validation failure (required field absent)
Command not on allowlist400validate_server_config() — only npx/node/python/python3/uvx/docker/deno/bun allowed
Command not found at spawn500npx / python not on PATH in subprocess env

Auto-Restart Policy

Servers can be configured to restart automatically on failure. The policy is set at add time and stored in the server config:

PolicyBehaviour
never (default)No automatic restart on failure
on-failureRestart on non-zero exit code
alwaysRestart on any termination

Restart delays use exponential backoff: delay = 5s × 2^min(restart_count, MAX_BACKOFF_EXPONENT) where MAX_BACKOFF_EXPONENT = 5, giving a maximum delay of 5 × 32 = 160 s. At restart count 0 (first crash), the delay is 5 s.

Concurrent Operation Safety

ServerManager uses a per-server asyncio.Lock to prevent concurrent Start/Stop/Restart operations on the same server. If a server is already being modified, a second request returns immediately with False (fast-fail). The lock is also used for tool calls (threading.Lock for the stdio read/write cycle).

localStorage Tool History

toolHistoryService stores the last 50 executions per server+tool combination in localStorage. This is entirely client-side — nothing is sent to the backend. Clearing browser storage wipes the history.

Common Debugging Patterns

Server won't start — "Init timeout after 30 s"

  • Check the server's stderr logs: GET /api/servers/:id/logs
  • Verify the command is on PATH: run it manually in a terminal
  • For SSE/HTTP servers: check the url field in metadata.json matches the port the server binds to

Env update fails with 400 "No valid environment variables"

  • All submitted values were empty or matched a placeholder pattern (e.g. YOUR_KEY_HERE)
  • Make sure actual values are set before submitting

Tool call returns error / no result

  • Confirm the server is in running state before calling tools
  • Check inputSchema — required parameters must be present in arguments
  • For concurrent calls: only one tool call runs at a time per server (serialised by lock); others queue behind it

Stop returns 403

  • The token you're using doesn't match the user who started the server
  • Use the same token, or run the server in --allow-insecure mode for anonymous access
